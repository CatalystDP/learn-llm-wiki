# RAG 全栈工程

> 高质量 RAG 不是“检索前加一步向量召回”，而是数据、检索、编排、评测、治理与基础设施的联合作业系统。

## 核心观点

1. **应用场景先于技术堆栈**：智能问答、文档助手、代码助手、研究分析和知识管理对应不同的检索粒度、时效性要求与答案约束。
2. **编排层把检索变成完整工作流**：查询理解、检索路由、重排、上下文组织和结果后处理共同决定 RAG 是否可用。
3. **数据处理是检索质量的上游决定因素**：采集、清洗、分块、向量化和索引构建决定了召回空间的形状。
4. **评测与运维闭环决定系统能否长期演进**：离线指标、链路追踪、A/B 测试、告警和成本分析让系统从“能跑”走向“可运营”。
5. **安全与合规应作为架构常量存在**：权限、脱敏、审计、模型安全和隐私保护必须嵌在全链路中，而不是上线前补一层网关。

- [[AI Agent 全栈工程]] — 当 Agent 需要知识密集推理时，RAG 往往成为其外部知识层与证据层。

## 分层落地顺序

1. 先选单一业务场景，明确回答格式、引用要求、时效性和风险边界。
2. 优先建设高质量数据处理管道，再调检索策略与上下文拼装。
3. 在放量前补齐离线评测、追踪、告警、权限和审计。
4. 只有当单跳检索无法满足复杂任务时，再引入多路召回、多阶段重排或 Agent 化编排。

## 未解决的问题

- 多模态检索在企业知识库里什么时候真正值得引入？
- 知识图谱增强应该作为独立层，还是只在高价值领域做局部补充？
- 离线评测指标如何与真实用户满意度、业务转化和成本目标对齐？
- RAG 编排和 Agent 编排的边界应该如何划分，何时合并、何时分离？
