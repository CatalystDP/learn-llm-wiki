# AI Agent 全栈工程

> 核心问题不是“选哪个模型”，而是如何把决策、执行、数据、治理与持续改进组合成可靠的生产系统。

## 核心观点

1. **应用价值定义系统边界**：从客服、知识助手、数据分析或研发辅助等小场景出发，先明确成功标准和允许执行的动作。
2. **Harness 与 Runtime 分离**：[[Agent Harness]]承载规划、状态和路由，[[Agent Runtime]]承载执行、适配、隔离和并发；这使策略演进不必绑死底层实现。
3. **能力与数据共同约束效果**：模型、工具、知识检索和数据治理共同决定 Agent 能知道什么、能做什么以及结果是否可信。
4. **横切能力决定生产成熟度**：[[Agent 可观测性]]解释行为，[[Agent 安全治理]]限制风险，[[Agent DevOps]]把反馈转成可验证的改进。
5. **架构图是检查表，不是采购清单**：应按业务风险和规模逐层补齐能力，避免为技术完整性引入不必要复杂度。

## 关键概念

- [[Agent Harness]] — 管理规划、工作流、路由、协作、状态和上下文。
- [[Agent Runtime]] — 执行模型与工具调用并提供适配、隔离和并发。
- [[Agent 可观测性]] — 连接技术运行信号、任务结果和用户反馈。
- [[Agent 安全治理]] — 约束身份、数据、网络、模型行为与审计。
- [[Agent DevOps]] — 形成从开发到线上反馈的持续迭代闭环。

## 分层落地顺序

1. 定义单一业务场景、评价指标和风险边界。
2. 构建最小 Harness 与 Runtime，只接入必要的模型、工具和知识。
3. 在扩大权限或流量前补齐追踪、评测、审计和安全控制。
4. 用真实失败与反馈推动模块化扩展，而不是一次性堆满技术栈。

## 未解决的问题

- 每一层的标准接口、状态所有权和故障边界如何定义？
- 哪些任务真的需要多智能体，收益如何抵消协调成本？
- 如何联合衡量任务质量、时延、成本、安全和业务价值？
- 高风险工具动作应在哪一层拦截，何时必须人工确认？
- 模型、提示、知识、工具和策略更新如何做可复现评测与回滚？
